## P4 — GitHub Actions 工程化（决定项目是否"稳如老狗"）

| # | 任务 | 说明 | 预期效果 |
|---|------|------|---------|
| 28 | **Actions 运行失败时自动发 Issue/邮件告警** | 当 Actions 运行失败（如 API 额度用完、SMTP 认证失败）时，自动在仓库创建一个 Issue 或发一封告警邮件 | 用户不用每天去 Actions 里检查 |
| 29 | **添加 API 请求重试 + 退避机制** | arXiv/LLM API 偶尔网络超时，用 `tenacity` 库自动重试 3 次，避免偶发失败导致当天无推送 | 提升稳定性 |
| 30 | **LLM 响应异常降级** | 当 LLM API 返回格式异常/超时时，降级到简单模式（如只推送标题+链接，不做摘要），保证"今天至少有推送" | 服务可用性优先 |
| 31 | **Actions 添加缓存** | 用 `actions/cache` 缓存 pip/npm 依赖，减少每次运行安装时间（当前每次都要重装） | 运行速度提升 30~50% |
| 32 | **运行报告持久化到仓库** | 每次运行后在仓库中生成 `reports/2026-04-24.json`，记录：推送了哪些论文、用了多少 token、耗时多久 | 方便用户追溯历史 |
| 33 | **配置版本校验与迁移提示** | 当 `config.yml` 结构升级时，在 Actions 日志中提示用户"请同步更新 config.yml，新增字段 xxx" | 避免升级后配置不兼容导致失败 |
| 34 | **敏感信息防泄漏检查** | 在 CI 中添加 `truffleHog` 或 `git-secrets`，防止用户误将 API Key 提交到代码中 | 安全最佳实践 |
